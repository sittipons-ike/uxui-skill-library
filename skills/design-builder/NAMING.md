# Token + File Naming Convention — design-system spec

## 0a. File architecture — 3-file split

Beanly-pattern split (the canonical setup):
```
design.md       → tokens (primitive + semantic + mood + iconography)
components.md   → atomic (atom + molecule + organism)
ui.md           → ui (page + pattern + section + flow)
```

Each file has a single scope. No file may carry blocks from another scope.

### Cross-file ref syntax — prefix style
All refs across files use a file-prefix:
```
{design.primitive.*}        → from design.md
{design.semantic.*}         → from design.md
{components.atom.*}         → from components.md
{components.molecule.*}     → from components.md
{components.organism.*}     → from components.md
{ui.page.*}                 → from ui.md
{ui.pattern.*}              → from ui.md
{ui.section.*}              → from ui.md
{ui.flow.*}                 → from ui.md
```

### Ref direction — strict downward
```
ui.md      ──refs──▶ components.md  +  (design.md for layout only)
                            │
                            ▼
                        design.md
```

| Allowed | Forbidden |
|---|---|
| ui → components, ui → design (layout only) | components → ui, design → ui/components |
| components → design | design → anything |
| Same-file refs (atom → atom, page → page slot reuse) | – |

**Upward ref = audit fail (Critical).**

### Scope rules — what may live where

| File | YAML root keys allowed |
|---|---|
| `design.md` | `primitive:`, `semantic:`, `mood:`, plus inline `iconography:` under `semantic:` |
| `components.md` | `component:` with `atom:`, `molecule:`, `organism:` nested |
| `ui.md` | `ui:` with `page:`, `pattern:`, `section:`, `flow:` nested |

A `component:` block in `design.md` = audit fail.
A `primitive:` block in `components.md` = audit fail.
A `semantic:` block in `ui.md` = audit fail.

### Frontmatter required keys per file

Every file must declare:
```yaml
brand: <Name>
version: <semver>
scope: 'tokens-only' | 'components-only' | 'ui-compositions'
last-updated: <YYYY-MM-DD>
```

`components.md` and `ui.md` also declare:
```yaml
depends-on:
  - './design.md'
  - './components.md'   # only for ui.md
```

### Single-file legacy mode (still supported)
Old monolithic `DESIGN.md` (all blocks in one file) is still valid. Audit detects:
- If 3 separate files exist → use split-architecture rules (above)
- If only 1 file exists (monolithic) → use legacy rules + show INFO: "Run split-migrate to adopt 3-file architecture"

Refs in monolithic mode use bare paths (`{semantic.*}`) — no prefix. Switching to split requires rewriting refs with prefix.

### Migration from monolithic → split

```bash
# 1. Backup the legacy file
mv DESIGN.md DESIGN.legacy.md

# 2. Run split-migrate (built into design-md-audit with --migrate flag)
#    OR manually split:
#      design.md      = primitive + semantic + mood + body sections (Overview/Mood/Primitives/Semantic/Layout/Do-Don't/...)
#      components.md  = component: block + ## Component Tokens body section, refs rewritten with {design.*} prefix
#      ui.md          = empty starter or built via design-ui-builder

# 3. Run audit to verify cross-file refs resolve
```

# Token Naming Convention — original (within-file rules)

> **Status:** Draft v1.0 — proposed standard for Primitive + Semantic + Component token layers.
> **Scope:** Applies to all DESIGN.md files generated by `design-builder` and audited by `design-md-audit`.

---

## 0. Why this exists

Every token in DESIGN.md is referenced by other tokens, components, and coding agents. Without a strict convention:
- Refs break when a token is renamed
- Designers and agents write the same idea differently (`text-primary` vs `textPrimary` vs `color.text.primary`)
- Merging multiple DESIGN.md files becomes manual work
- Tailwind / CSS / Figma exports require ad-hoc translation

This spec locks every decision so token paths are deterministic across the whole pipeline.

---

## 1. Token reference syntax

### 1.1 Path separator: **dot** (`.`)
```
✅  {primitive.colors.red.500}
✅  {semantic.colors.text.primary.default}
❌  {primitive/colors/red/500}            # Figma-style slash — banned in DESIGN.md
❌  {primitive-colors-red-500}            # dash — confused with kebab-case keys
❌  primitive.colors.red.500              # missing braces in body text
```

### 1.2 Braces are required in body text
- Inside YAML frontmatter values: `default: '{primitive.colors.red.500}'`
- Inside markdown body: `The CTA uses {semantic.colors.primary.default}`
- Inside code blocks: keep `{...}` so refs stay grep-able

### 1.3 No partial paths
Always go from root: `primitive.*`, `semantic.*`, or `component.*`. No `colors.red.500` shorthand.

---

## 2. Key naming rules

### 2.1 Case: **kebab-case** for compound keys
```yaml
✅  line-height: 24
✅  border-width: 1
✅  text-primary:
✅  bg-default:
❌  lineHeight: 24            # camelCase
❌  line_height: 24           # snake_case
❌  LineHeight: 24            # PascalCase
```

### 2.2 Single-word keys: bare
```yaml
✅  red:
✅  spacing:
✅  size:
❌  red-hue:                  # don't pad with -hue
❌  spacing-scale:            # don't pad with -scale
```

### 2.3 Reserved top-level keys
Only these three allowed at the YAML root:
```yaml
primitive:
semantic:
component:    # Phase 3 — leave out until you reach component tier
```

## 3. Scale stop conventions

### 3.1 Primitive layer: **numeric** only
```yaml
primitive:
  colors:
    red:
      50: '#fef2f2'
      500: '#ef4444'
      950: '#450a0a'
  spacing:
    0:  0
    4:  4
    16: 16
  radius:
    8:    8
    9999: 9999    # pill marker stays numeric in primitive
```

**Why:** primitives are raw — they have no abstract meaning. `red.500` describes WHAT it is (a hex on the red ramp), not HOW it's used.

### 3.2 Semantic layer: **t-shirt** only
```yaml
semantic:
  typography:
    body:
      sm:  { size: '{primitive.typography.size.sm}',  ... }
      md:  { ... }
      lg:  { ... }
  spacing:
    xs:  '{primitive.spacing.4}'
    sm:  '{primitive.spacing.8}'
    md:  '{primitive.spacing.16}'
  radius:
    sm:   '{primitive.radius.4}'
    md:   '{primitive.radius.8}'
    lg:   '{primitive.radius.16}'
    pill: '{primitive.radius.9999}'
```

**Allowed t-shirt stops** (in order):
```
2xs, xs, sm, md, lg, xl, 2xl, 3xl, 4xl, 5xl, 6xl, 7xl, 8xl, 9xl
+ special: pill, full, none
```

### 3.3 Forbidden: mixing
```yaml
❌  primitive:
      spacing:
        sm: 8                  # t-shirt in primitive — banned
❌  semantic:
      radius:
        500: 8                 # numeric in semantic — banned
```

## 4. Color hue naming

### 4.1 Tailwind hues: **exact name, no aliasing**
Allowed (Tailwind v3.4+):
```
slate, gray, zinc, neutral, stone,
red, orange, amber, yellow, lime,
green, emerald, teal, cyan, sky,
blue, indigo, violet, purple, fuchsia,
pink, rose
```

```yaml
✅  red:
✅  cyan:
❌  crimson:                   # Tailwind doesn't have this — use red or custom
❌  gray-blue:                 # not a Tailwind name
❌  primary-red:               # don't pad — `red` is just `red`
```

### 4.2 Custom hue: bare single word + provenance comment
When Tailwind doesn't match the brand color (e.g., a very specific brand hex):
```yaml
primitive:
  colors:
    brand:                     # custom — not in Tailwind
      50:  '#fff5f7'
      500: '#e30b5d'           # exact brand hex
      950: '#3a0218'
      # custom-source: 'Brand Guide 2026-05-12, hand-tuned by lead designer'
```

**Rules for custom:**
- Single word, no prefix (`brand`, `lotto`, `midnight` — not `acme-red`, `app-brand`)
- Must include a comment line: `# custom-source: <where these came from>`
- Must define all 11 stops (50–950) even if some are interpolated
- One custom hue per project unless multi-brand

### 4.3 Banned hue names
```
apple-blue, spotify-green, airbnb-rausch, claude-warm
                                                     ^ brand-suffix patterns
neutral-1, neutral-2                ^ numbered suffixes
```

## 5. Modifiers

### 5.1 Opacity: `@<n>%`
```
{primitive.colors.neutral.900}@10%
{primitive.colors.neutral.900}@80%
```
- Suffix only — never prefix
- Only on color refs
- Resolves to `rgba(...)` at build time
- Allowed steps: 0, 5, 10, 20, 40, 60, 80, 100 (must match `primitive.opacity`)

### 5.2 State: `.<state>` (**component layer only — Phase 3**)
States belong to component tier, NOT semantic. Semantic uses scale stops; component maps state→stop.

```
✅  {component.button.primary.bg.hover}    # component owns state
✅  {component.button.primary.bg.hover} → resolves to {semantic.colors.primary.dark}
❌  {semantic.colors.primary.hover}        # semantic has no states
❌  {primitive.colors.red.hover}           # primitive never has state
```

**Allowed states** (exact list):
```
default, hover, focused, pressed, disabled, active, selected
```

State is the LAST segment of a component path.

**Semantic uses scale instead of state:**
```
semantic.colors.primary:
  default      ← used at component.*.default
  darker       ← used at component.*.pressed
  dark         ← used at component.*.hover
  light        ← used at component.*.disabled (subtle)
  soft-light   ← used at component.*.disabled (background)
```
The component decides which scale stop maps to which state.

### 5.3 Mode: YAML key, **not** in path
Modes (light/dark/high-contrast) are top-level keys inside the semantic block, not part of token paths:
```yaml
semantic:
  colors:
    text:
      primary:
        default:
          light: '{primitive.colors.neutral.900}'
          dark:  '{primitive.colors.neutral.50}'
```

Refs from body text drop the mode:
```
Card text uses {semantic.colors.text.primary.default}
```
The runtime resolves mode based on the active theme.

## 6. Component-scoped tokens (Tier 3) — Atomic Design + Alias

### Architecture: 3-tier atomic

```
atom        → smallest indivisible (button, input, icon, badge, label, avatar)
molecule    → atoms composed (form-field, nav-item, search-bar, stat-tile)
organism    → molecules composed (sidebar, topbar, table, hero)
```

(template + page tiers exist in Brad Frost's full Atomic Design but live in the page-layer / app code, NOT in DESIGN.md.)

### Path syntax — atomic 3-tier
```
{component.<tier>.<name>.<variant>.<prop>.<state>}
                ↑     ↑       ↑       ↑      ↑
            atom    button  primary  background rest
```

Examples:
```
{component.atom.button.primary.background.rest}
{component.atom.button.primary.background.hover}
{component.atom.input.text.border.focus}
{component.molecule.form-field.error.helper.foreground.rest}
{component.organism.sidebar.default.background.rest}
```

**Tier is the first segment after `component.`** — non-negotiable order: `component → tier → name → variant → prop → state`.

### Prop names — one vocabulary, every layer (v6.3+)

**No vocabulary swap.** The same word is used in `design.md`, `components.json`, generated `tokens.css`, and the final HTML/CSS. Previous versions renamed props between spec and runtime (`surface`→`bg`, `content`→`fg`, etc.) — that indirection caused drift between what a designer read in the spec and what a dev saw in the shipped code. Removed.

Naming is based on **shadcn/ui** (the current React/Tailwind convention most devs already know), with gaps filled from the **W3C DTCG** composite-type vocabulary:

| Prop | Source | Meaning |
|---|---|---|
| `background` | shadcn (`--background`) | container fill |
| `foreground` | shadcn (`--foreground`) | text + icon color |
| `border` | shadcn (`--border`) / DTCG composite type | outline / stroke |
| `shadow` | DTCG composite type (shadcn has no shadow token) | depth |
| `ring` | shadcn (`--ring`) | accessibility focus indicator |
| `padding-x/y` | CSS native | inner spacing |
| `radius` | CSS native (`border-radius`) | shape softness |
| `text-style` | composed type (family+size+line-height+weight+tracking together) | composed type style |
| `gap` | CSS native (flex/grid `gap`) | space between children |
| `width/height` | CSS native | overall size |

**Legacy note:** designs generated before this rename used `surface`/`content`/`edge`/`elevation`/`focus-halo` as **component prop names**. Treat those as deprecated synonyms of `background`/`foreground`/`border`/`shadow`/`ring` — migrate on next edit, don't leave both vocabularies in the same file.

⚠️ **Do not confuse with the semantic color role `surface`** (§ Color roles: `surface.base/raised/sunken` — a Material-3-style color category, e.g. `{semantic.colors.surface.raised}`). That role name is unrelated to this prop-naming fix and stays as `surface` — only the **component prop** (the generic "fill" property on any atom/molecule/organism) is renamed to `background`.

### State aliases — interaction stage names

| Old name | New alias | Meaning |
|---|---|---|
| `default` | **`rest`** | resting / idle (industry term) |
| `hover` | `hover` | (kept — universal) |
| `pressed` | **`active`** | CSS-aligned (`:active`) |
| `focused` | **`focus`** | shorter, CSS-aligned (`:focus`) |
| `disabled` | `disabled` | (kept) |
| `active` (was: nav current) | **`selected`** | chosen item / current page |
| `selected` (was: list pick) | `selected` | (kept — already canonical) |
| `error` | `error` | validation only |

**Important rename note:** old `pressed` → `active`. Old `active` (nav current page) → `selected`. This eliminates the ambiguity where `active` meant 2 different things.

### Allowed states (canonical list — final)
```
rest, hover, active, focus, disabled, selected, error
```
- `rest` — idle, no interaction
- `hover` — mouse over (or equivalent)
- `active` — pressed / engaged (CSS `:active`)
- `focus` — keyboard focus (CSS `:focus`)
- `disabled` — non-interactive
- `selected` — chosen / current (nav item, option, tab)
- `error` — validation failure (input only)

### Allowed props per atom class

| Atom class | Mandatory | Optional |
|---|---|---|
| button / chip | background, foreground | border, shadow, ring |
| input / textarea / select | background, foreground, border | ring, helper, placeholder |
| icon | foreground | extent |
| badge | background, foreground | border |
| label | foreground | – |
| avatar | background, foreground | border, extent |

### Molecule structure

```yaml
molecule:
  form-field:
    composed-of:
      label: '{component.atom.label.default}'
      input: '{component.atom.input.text}'
      helper: '{component.atom.helper-text.default}'
    layout:
      stack-gap: '{semantic.spacing.xs}'
    error:                          # state override at molecule level
      label.foreground.rest:  '{semantic.colors.text.state.error}'
      input.border.rest:      '{semantic.colors.border.error}'
      helper.foreground.rest: '{semantic.colors.text.state.error}'
```

`composed-of` lists atoms used.
`layout` defines internal layout tokens.
State overrides at molecule level only when the visual differs from atoms' own states.

### Organism structure

```yaml
organism:
  sidebar:
    composed-of:
      brand-mark: '{component.atom.brand-mark.default}'
      nav-section: '{component.molecule.nav-section.default}'
      user-pill: '{component.molecule.user-pill.default}'
    layout:
      extent-x: 240
      background.rest: '{semantic.colors.surface.raised}'
      border.rest: '{semantic.colors.border.tertiary}'
      inset: '{semantic.spacing.lg}'
      stack-gap: '{semantic.spacing.xl}'
```

### Sizes block — still per-atom root
```yaml
atom:
  button:
    sizes:
      md:
        inset-x: '{semantic.spacing.md}'
        inset-y: '{semantic.spacing.sm}'
        corner: '{semantic.radius.md}'
        text-style: '{semantic.typography.label.md}'
    primary:
      background:
        rest: '...'
        hover: '...'
        active: '...'
        focus: '...'
        disabled: '...'
```

Sizes never carry per-state values. State only attaches to interactive props.

### Literal values allowed
- `'transparent'` (no fill)
- `'none'` (no shadow / border / ring)
- `'slot'` (organism/template placeholder for content)

Everything else MUST be a `{semantic.*}` or `{component.atom.*}` / `{component.molecule.*}` ref.

### Forbidden in component layer
- Primitive refs (`{primitive.colors.red.500}`)
- Raw hex / px
- Deprecated prop names as **component props** (`surface`, `content`, `edge`, `elevation`, `focus-halo` — use `background`/`foreground`/`border`/`shadow`/`ring`). Does not apply to the semantic color role `surface` (`{semantic.colors.surface.*}`) — that's a different, unrelated token category and keeps its name.
- Old state names (`default`, `pressed`, `focused`)
- Old conflicting `active` for "current page" — use `selected`
- `disabled` modeled as a variant name instead of a state (real-world anti-pattern — seen in production apps with a `BUTTON_VARIANT.disabled` entry) — `disabled` is a state on every variant, never its own variant
- Tier mixing (organism → atom is allowed; atom → organism is forbidden — no upward refs)

## 7. Special cases

### 7.1 Gradients
```yaml
primitive:
  colors:
    brand:
      gradient-1: 'linear-gradient(135deg, #e30b5d 0%, #ff385c 100%)'
```
- Gradient definitions go in primitive
- Use kebab key `gradient-<n>` where `<n>` is a number, not a name
- Semantic layer refs the whole gradient: `{primitive.colors.brand.gradient-1}`

### 7.2 Compound shadows (2-layer)
```yaml
primitive:
  shadow:
    md:
      - { x: 0, y: 1, blur: 2,  spread: 0, color: '{primitive.colors.neutral.900}@10%' }
      - { x: 0, y: 4, blur: 8,  spread: 0, color: '{primitive.colors.neutral.900}@8%'  }
```
YAML array = multi-layer. Single object = single layer.

### 7.3 Typography composition
Semantic layer composes atomic primitives into roles:
```yaml
semantic:
  typography:
    heading:
      h1:
        family:      '{primitive.typography.family.sans}'
        size:        '{primitive.typography.size.5xl}'
        line-height: '{primitive.typography.line-height.5xl}'
        weight:      '{primitive.typography.weight.bold}'
        tracking:    '{primitive.typography.tracking.tight}'
```
Never inline numbers in semantic typography — always ref.

**Optional responsive variance (v6.4+):** a role MAY add a `responsive:` sibling key with diff-only overrides per breakpoint tier (mobile-first cascade — last declared tier at or below the viewport wins):
```yaml
semantic:
  typography:
    heading:
      h1:
        family:      '{primitive.typography.family.sans}'
        size:        '{primitive.typography.size.5xl}'       # base — used at every breakpoint unless overridden below
        line-height: '{primitive.typography.line-height.5xl}'
        weight:      '{primitive.typography.weight.bold}'
        tracking:    '{primitive.typography.tracking.tight}'
        responsive:
          sm: { size: '{primitive.typography.size.4xl}', line-height: '{primitive.typography.line-height.4xl}' }
          lg: { size: '{primitive.typography.size.6xl}', line-height: '{primitive.typography.line-height.6xl}' }
```
- `responsive:` is optional per role — omit for roles that don't need it (most `body`/`label`/`caption` roles never do)
- Tier keys must be from `semantic.breakpoints` (§ Breakpoint roles) — no new tier names invented here
- Each tier is diff-only — list only the sub-keys that change from base
- A `design.md` with no `responsive:` key is unaffected — fully backward compatible, no migration required on existing files

## 8. Forbidden patterns (summary)

| Pattern | Reason |
|---|---|
| `{primitive/colors/red/500}` | wrong separator |
| `primitive.colors.red.500` (no braces) | not parseable |
| `lineHeight`, `line_height` | wrong case |
| `apple-blue`, `spotify-green` | brand-suffix |
| `red-hue`, `spacing-scale` | redundant padding |
| `red-500` as key | use nested `red: { 500: ... }` |
| Mixing numeric and t-shirt stops in one layer | breaks tier model |
| State in primitive (`primitive.colors.red.hover`) | states are semantic concepts |
| Raw hex in semantic block | semantic must ref primitive |
| Raw hex in markdown body | body must use `{...}` refs |
| `surface`/`content`/`edge`/`elevation`/`focus-halo` used as a **component prop** name | deprecated — use `background`/`foreground`/`border`/`shadow`/`ring` (shadcn + DTCG vocab, no swap between spec and runtime). The semantic color role `surface` is unaffected. |
| `disabled` as a variant name (instead of a state) | `disabled` must be a state on every variant, never its own variant |

## 9. Migration from existing DESIGN.md

When importing/remixing a DESIGN.md that uses other conventions, normalize in this order:
1. Brand-suffix hues → drop suffix (`airbnb-rausch` → `brand` with provenance comment)
2. camelCase keys → kebab-case
3. Inline hex in semantic → extract to primitive, replace with ref
4. State as separate token (`primary-hover`) → nest under semantic state (`primary.hover`)
5. Mode as path suffix (`primary-dark`) → move to mode key

## 10. Validation checklist

`design-md-audit` enforces:
- [ ] All token refs use `{primitive.*}`, `{semantic.*}`, or `{component.*}`
- [ ] All keys are kebab-case or single bare word
- [ ] Primitive uses only numeric stops; semantic uses only t-shirt stops
- [ ] All Tailwind-named hues match Tailwind v3.4+ hex exactly
- [ ] Custom hues have `# custom-source:` comment + all 11 stops
- [ ] States appear only in semantic/component paths, only as last segment
- [ ] Opacity modifier `@n%` only on color refs, only with allowed steps
- [ ] No raw hex/px in markdown body sections

## 12. Known Gaps entry format (v6.0.0+)

When `design-builder` (or any downstream skill) needs to log an auto-shift away from a client-given input or a contrast-fail fallback, use this **exact format** in the `## Known Gaps` section of `design.md`:

```markdown
## Known Gaps

- shifted from client-provided #XXXXXX → semantic.<role>.<state> = #YYYYYY (reason: <short reason>, original ratio: A.A:1, shifted ratio: B.B:1)
- mapped from client #XXXXXX → tailwind.<hue>.<stop> (reason: ΔE < 5)
- custom hue generated from client #XXXXXX → primitive.colors.brand.50..950 (reason: ΔE ≥ 5, no Tailwind match)
- TBD: <free-form note for future work>
```

**Rules:**
- One bullet per gap. No nested lists.
- The `shifted from client-provided` keyword is **reserved** — only used when honoring client-given palette (Step 1d). `design-md-audit` treats these as informational, not errors.
- `mapped from client` and `custom hue generated from client` are also reserved keywords — both informational.
- `TBD:` prefix is free-form, treated as audit warnings.
- Hex values lowercase, 6 chars. Reasons under 80 chars.

`design-md-audit` parses this section by keyword prefix. New keywords require parser update.
