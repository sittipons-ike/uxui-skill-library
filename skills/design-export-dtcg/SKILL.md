# design-export-dtcg

Cross-platform token export. Convert your design system's `design.md` (semantic + primitive tiers) and optional `components.json` (comp aliases) into a **W3C DTCG-compliant `tokens.json`**, paired with a **Style Dictionary** config so the team can compile a single source of truth into web (CSS/SCSS/TS), iOS (Swift), Android (Kotlin), Flutter (Dart), and Tailwind.

This skill is the bridge between the design system spec (markdown/JSON) and native platform code.

---

## When to use

- Team is starting a **mobile/native** project (iOS Swift, Android Compose, Flutter) that must share tokens with the existing web DS.
- You need a **single source of truth** across web + native — no more re-typing hex codes in three places.
- You want to use the **Style Dictionary** ecosystem (Amazon's tooling) and its plugin/transform ecosystem.
- Your DS has reached **production maturity** (v6 stable) and the semantic layer is no longer churning weekly.

---

## When NOT to use

- DS is **web-only** — `tokens.css` emitted by `/design-component-builder` is already enough.
- DS is **still iterating** rapidly — exporting now means re-exporting every day. Wait for a stable semantic freeze.
- You don't have a real consumer for `tokens.json` yet — don't generate dead artifacts.

| File | Required | Purpose |
|---|---|---|
| `./design.md` | required | Source of `primitive:` + `semantic:` token blocks |
| `./components.json` | optional | Source of comp-tier aliases (`atom.*.tokens`) — emitted as DTCG aliases pointing at semantic |

## DTCG Format

This skill emits the **W3C Design Tokens Community Group** draft spec format. Every token has `$value` + `$type` (required), and may have `$description`.

### Concrete value example

```json
{
  "color": {
    "primary": {
      "default": {
        "$value": "#2563eb",
        "$type": "color",
        "$description": "Primary action color — buttons, links, focus"
      }
    }
  }
}
```

### Alias example (matches our `{ref}` pattern)

DTCG aliases use **brace syntax** — same shape as our internal `{semantic.color.primary.default}` refs, so the migration is mechanical:

```json
{
  "btn": {
    "bg": {
      "$value": "{color.primary.default}",
      "$type": "color",
      "$description": "Used by atom.button — primary background"
    }
  }
}
```

A token file may mix concrete and alias values — primitives stay concrete, semantic resolves to primitive, comp resolves to semantic.

## Execution Steps

### Step 1 — Read `design.md`

- Parse the `primitive:` block (e.g. `primitive.color.blue.500: "#2563eb"`).
- Parse the `semantic:` block (e.g. `semantic.color.primary.default: "{primitive.color.blue.500}"`).
- Capture token paths, values, refs, and any inline descriptions.

### Step 2 — Read `components.json` (if present)

- For each entry under `atom.*`, `molecule.*`, `organism.*`, read the `tokens` block.
- Each comp token becomes a DTCG alias pointing at a semantic token — e.g. `atom.button.tokens.bg: "{semantic.color.primary.default}"` becomes:
  ```json
  "comp": { "button": { "bg": { "$value": "{color.primary.default}", "$type": "color" } } }
  ```

### Step 3 — Build `tokens.json` structure

- Group by category at the top level: `color`, `radius`, `space`, `shadow`, `font`, `duration`, `border`, etc.
- For each **concrete** token: emit `{ "$value": "<hex|px|...>", "$type": "<dtcg-type>", "$description": "..." }`.
- For each **alias** token: emit `{ "$value": "{path.to.target}", "$type": "<dtcg-type>", "$description": "Used by <component>" }`.
- Drop the `primitive.` / `semantic.` / `comp.` prefix from the brace path — DTCG resolves relative to the JSON root.

### Step 4 — Write `tokens.json`

- Top-level metadata block:
  ```json
  {
    "$schema": "https://design-tokens.github.io/community-group/format/tokens.schema.json",
    "$description": "Generated by design-export-dtcg v1.0.0. Source: design.md + components.json. Do not edit by hand — re-run /design-export-dtcg."
  }
  ```

### Step 5 — Write `style-dictionary.config.js`

Template covering web + iOS + Android + Flutter + Tailwind:

```js
// style-dictionary.config.js — generated by /design-export-dtcg
module.exports = {
  source: ['tokens.json'],
  platforms: {
    css:      { transformGroup: 'css',     buildPath: 'build/web/',     files: [{ destination: 'tokens.css',  format: 'css/variables' }] },
    scss:     { transformGroup: 'scss',    buildPath: 'build/web/',     files: [{ destination: '_tokens.scss', format: 'scss/variables' }] },
    ts:       { transformGroup: 'js',      buildPath: 'build/web/',     files: [{ destination: 'tokens.ts',   format: 'javascript/es6' }] },
    ios:      { transformGroup: 'ios-swift', buildPath: 'build/ios/',   files: [{ destination: 'Tokens.swift', format: 'ios-swift/class.swift', className: 'Tokens' }] },
    android:  { transformGroup: 'android', buildPath: 'build/android/', files: [{ destination: 'Tokens.kt',   format: 'compose/object', className: 'Tokens', packageName: 'com.app.tokens' }] },
    flutter:  { transformGroup: 'flutter', buildPath: 'build/flutter/', files: [{ destination: 'tokens.dart', format: 'flutter/class.dart', className: 'Tokens' }] },
    tailwind: { transformGroup: 'js',      buildPath: 'build/web/',     files: [{ destination: 'tailwind.tokens.js', format: 'javascript/module' }] }
  }
};
```

## Constraints

- Skill writes **ONLY** `tokens.json` + `style-dictionary.config.js`. Nothing else.
- Does **NOT** run `style-dictionary build` — user does that.
- Does **NOT** install npm packages — user does that.
- **Idempotent** — re-running overwrites both files with the current `design.md` + `components.json` state. No merge, no diff prompts.
- Refuses to run if `design.md` is missing or has no `semantic:` block (asks user to run `/design-builder` first).

---

## Validation

- `tokens.json` must validate against the **W3C DTCG draft schema** (referenced in `$schema`).
- All refs use `{dot.path}` format — matches DTCG alias syntax exactly (no migration needed).
- `$type` values must come from the canonical DTCG type list:
  - `color`, `dimension`, `fontFamily`, `fontWeight`, `duration`, `cubicBezier`, `number`, `string`, `typography`, `shadow`, `border`
- Any unknown type → skill stops and reports the offending token path.

---

## Cross-platform sync workflow

This is the steady-state loop once the DS is exported:

1. Designer edits `design.md` (e.g. tweaks `semantic.color.primary.default`).
2. Run `/design-component-builder` → updates `tokens.css` (web preview).
3. Run `/design-export-dtcg` → updates `tokens.json`.
4. CI (or local dev) runs `npx style-dictionary build` → all platform files re-emit.
5. Web + iOS + Android + Flutter all pick up the new value on next build.

The DS spec stays the single source of truth. Every platform consumes a generated artifact — none of them hand-roll hex codes.
