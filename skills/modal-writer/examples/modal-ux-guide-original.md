# Modal UX Writing Guide v1.2
## ระบบหลังบ้าน Lottery Plus Back-Office

> **วิธีใช้ไฟล์นี้:** แนบไฟล์นี้พร้อมกับ prompt เช่น:
> _"อ้างอิงจาก guide นี้ ช่วยเขียนข้อความ modal สำหรับ [ชื่อ feature] ที่มี action คือ [...]"_

---

## 1. โทนและภาษา (Tone of Voice)

**Platform:** Desktop Browser เท่านั้น — ใช้ "คลิก" ไม่ใช่ "กด"

| หลักการ | ความหมาย |
|---|---|
| ทางการแต่เข้าใจง่าย | ภาษาราชการระดับกลาง ไม่ใช้ภาษาพูดหรือศัพท์เฉพาะ |
| กระชับ ตรงประเด็น | ประโยคสั้น บอกสาระสำคัญภายใน 1–2 ประโยค |
| เน้นการกระทำ | บอกผู้ใช้ว่าต้องทำอะไร ไม่อธิบายกระบวนการภายใน |
| สอดคล้องกัน | ใช้คำเดิมในความหมายเดิมทั้งระบบ ไม่สลับคำ |

## 2. ประเภท Modal และ Intent

| ประเภท | Icon | สี | Intent | Tone |
|---|---|---|---|---|
| Confirmation ทั่วไป | ⚠️ สามเหลี่ยม filled | เหลือง/ส้ม | ยืนยัน action ที่ reversible | Neutral / Serious |
| Confirmation Destructive | △ สามเหลี่ยม outline บนพื้นแดง | แดง | ยืนยัน action ที่ย้อนกลับไม่ได้ | Serious / Warning |
| Success | ✓ วงกลม + ติ๊กถูก | เขียว #16A34A | แจ้งว่าดำเนินการสำเร็จ | Positive |
| Error / Failure | ✕ วงกลม + X | แดง | แจ้งว่าล้มเหลว + next step | Calm / ไม่ตำหนิ |
| Info / Notice | ℹ วงกลม + i | น้ำเงิน | ให้ข้อมูลก่อนดำเนินการ | Neutral / Informative |
| Loading | ⟳ Spinner animated | น้ำเงิน | แสดงว่าระบบกำลังประมวลผล | Neutral |

**เกณฑ์เลือก Confirmation Icon:**
- ⚠️ เหลือง (1a) = Action **ย้อนกลับได้** หลัง confirm เช่น แก้ไข, วางขาย, อนุมัติ
- △ แดง (1b) = Action **ย้อนกลับไม่ได้เลย** เช่น ลบถาวร, ยกเลิกออเดอร์

## 3. โครงสร้าง Modal มาตรฐาน

```
┌─────────────────────────────────────────┐
│  [Icon]  Title (กริยา + นาม)            │
│                                         │
│  Body text — ผลที่จะเกิดขึ้น (≤2 บรรทัด) │
│                                         │
│        [ ยกเลิก ]  [ Primary CTA ]     │
└─────────────────────────────────────────┘
```

### กฎ Title
- ≤ 40 ตัวอักษร
- รูปแบบ **"กริยา + นาม"** เช่น "ลบรายการนี้", "วางขายสินค้า", "อนุมัติสลิป"
- ❌ ห้ามใช้ "ยืนยัน" นำหน้า (Modal เองคือ confirmation step)
- ❌ ห้ามใช้เครื่องหมายคำถาม (?)
- ระบุ Object ชัดเจน เช่น "ลบรายการนี้" ไม่ใช่แค่ "ลบ"

### กฎ Body
- ≤ 80 ตัวอักษร (ไม่เกิน 2 บรรทัด)
- **Destructive:** เน้นผลที่ย้อนกลับไม่ได้ — "รายการนี้จะถูกลบและไม่สามารถกู้คืนได้"
- **Confirmation ทั่วไป:** บอกผลที่เกิดขึ้น ไม่ต้องเน้น consequence หนัก
- **Success:** บอกว่าทำอะไรสำเร็จ กระชับ ≤ 1 ประโยค
- **Error:** อะไรเกิดขึ้น + ทำไม (ถ้าบอกได้) + ทำอะไรต่อ — ❌ ห้ามตำหนิผู้ใช้
- **Loading:** ไม่จำเป็นต้องมี Body หรือใช้ "กรุณารอสักครู่"

### กฎ Button
- Primary CTA ต้อง **mirror Action ใน Title** เสมอ
- ❌ ห้ามใช้ "ยืนยัน" เป็น CTA จริง (ใช้ verb เฉพาะ)
- ปุ่มไม่เกิน 2 ปุ่ม

## 4. Button Layout

### Layout A — Double CTA
```
[ ยกเลิก (outline) ]  [ Primary Action (filled) ]
```
**ใช้เมื่อ:**
- Destructive action (ลบ, ยกเลิกออเดอร์) — **บังคับ**
- Action มีผลกระทบต่อผู้อื่น (อนุมัติ, วางขาย)
- ต้องการให้ผู้ใช้พิจารณาก่อนตัดสินใจ
- Error modal ที่มี retry option

### Layout B — Single CTA
```
[ ════ Primary Action (full-width) ════ ]
            ยกเลิก (text link)
```
**ใช้เมื่อ:**
- Non-destructive action ทั่วไป
- Action ที่ผู้ใช้ต้องการ ~90%+ ของเวลา
- Success / Info modal ที่ต้องการ dismiss

## 5. Template สำเร็จรูปตามประเภท

### 5.1 Confirmation ทั่วไป ⚠️
```
Title:     [กริยา + นาม] เช่น "วางขายสินค้านี้"
Body:      [ผลที่จะเกิดขึ้น — บอกเล่า ไม่ตำหนิ]
Primary:   [กริยาเดียวกับ Title] เช่น "วางขาย"
Secondary: ยกเลิก
Layout:    Double หรือ Single
```

**ตัวอย่าง:**
| Title | Body | Primary | Secondary |
|---|---|---|---|
| วางขายสินค้านี้ | สินค้าจะแสดงบนหน้าขายทันที ผู้ซื้อเห็นได้ทันที | วางขาย | ยกเลิก |
| อนุมัติสลิปนี้ | ยอดที่ต้องชำระ: 945.00 บาท | อนุมัติ | ยกเลิก |
| ตั้งค่า Highlight | ลอตเตอรี่นี้จะแสดงในส่วน Highlight บนหน้าหลัก | ตั้งค่า | ยกเลิก |

---

### 5.2 Confirmation Destructive △ (แดง)
```
Title:     [กริยา + นาม] เช่น "ลบรายการนี้"
Body:      [ผลที่ย้อนกลับไม่ได้ — ชัดเจน]
Primary:   [กริยาเดียวกับ Title] — สีแดง
Secondary: ยกเลิก
Layout:    Double (บังคับ)
```

**ตัวอย่าง:**
| Title | Body | Primary | Secondary |
|---|---|---|---|
| ลบรายการนี้ | รายการนี้จะถูกลบออกจากระบบ และไม่สามารถกู้คืนได้ | ลบ | ยกเลิก |
| ยกเลิกออเดอร์นี้ | ออเดอร์นี้จะถูกยกเลิกและไม่สามารถดำเนินการต่อได้ | ยกเลิกออเดอร์ | ย้อนกลับ |

---

### 5.3 Success ✓ (เขียว)
```
Title:    [สิ่งที่สำเร็จ] เช่น "บันทึกข้อมูลสำเร็จ"
Body:     [กระชับ ≤1 ประโยค — optional]
Primary:  ตกลง (ปิดธรรมดา) / [action] เช่น คัดลอก (มี next step)
Secondary: — หรือ ย้อนกลับ (กรณีมี next step)
Layout:   Single (ปิดธรรมดา) / Double (มี next step)
```

---

### 5.4 Error / Failure ✕ (แดง)
```
Title:    ไม่สามารถ[action]ได้ หรือ [action]ไม่สำเร็จ
Body:     [อะไรเกิดขึ้น] + [ทำไม ถ้าบอกได้] + [ทำอะไรต่อ]
```

| กรณี | Primary | Secondary | Layout |
|---|---|---|---|
| Non-recoverable | ตกลง | — | Single |
| Recoverable | ลองอีกครั้ง | ยกเลิก | Double |

**Empathy Language:**
| ❌ ก่อน | ✅ หลัง |
|---|---|
| เกิดข้อผิดพลาดในการประมวลผลคำขอของท่าน | ไม่สามารถดำเนินการได้ในขณะนี้ |
| ระบบไม่สามารถ process request ได้ | เครือข่ายขัดข้อง กรุณาตรวจสอบการเชื่อมต่อ |

---

### 5.5 Info / Notice ℹ (น้ำเงิน)
```
Title:    [หัวข้อข้อมูล]
Body:     [รายละเอียดที่จำเป็น]
Primary:  รับทราบ
Layout:   Single หรือไม่มีปุ่ม
```

---

### 5.6 Loading ⟳ (น้ำเงิน) — ใหม่ v1.2
```
Title:    [optional] เช่น "กำลังดำเนินการ..."
Body:     กรุณารอสักครู่ (หรือไม่มี)
Button:   ไม่มีปุ่ม
```

## 6. คำมาตรฐาน (Standard Vocabulary)

### คำสำหรับปุ่ม (Button Labels)

| สถานการณ์ | ✅ ใช้ | ❌ ห้ามใช้ |
|---|---|---|
| Desktop / Web app | คลิก | กด, แตะ |
| Primary CTA ทั่วไป | [กริยาเฉพาะ] เช่น ลบ, อนุมัติ, วางขาย | ยืนยัน, โอเค |
| Secondary / dismiss | ยกเลิก | ปิด, ไม่, ย้อนกลับ (กรณีทั่วไป) |
| Error non-recoverable | ตกลง | โอเค, ปิด |
| Error recoverable | ลองอีกครั้ง | Retry |
| Info / Notice | รับทราบ | โอเค, ตกลง |

### คำสำหรับ Title

| ❌ ก่อนแก้ | ✅ หลังแก้ | เหตุผล |
|---|---|---|
| ยืนยันการวางขายสินค้า | วางขายสินค้านี้ | "ยืนยัน" ซ้ำซ้อนกับ UI pattern |
| ยืนยันอนุมัติสลิป | อนุมัติสลิปนี้ | กระชับ ชัดกว่า |
| ต้องการลบรายการนี้หรือไม่? | ลบรายการนี้ | ห้ามใช้ประโยคคำถาม |
| ยืนยันการตั้งค่า Highlight | ตั้งค่า Highlight | ตรงประเด็น |

## 7. กฎเด็ดขาด — ห้ามทำโดยไม่มีข้อยกเว้น

1. ❌ ห้ามใช้ "ยืนยัน" เป็น CTA จริง — Figma ใช้เป็น placeholder เท่านั้น ต้องแทนที่ก่อน implement
2. ❌ ห้ามใช้ "ยืนยัน" นำหน้า Title
3. ❌ ห้ามใช้ประโยคคำถามใน Title (?)
4. ❌ ห้ามใช้ปุ่ม "โอเค" — ใช้ "ตกลง" หรือ "รับทราบ"
5. ❌ ห้ามใช้คำว่า "กด" ใน Body — ใช้ "คลิก"
6. ❌ ห้าม Body ยาวเกิน 2 บรรทัด (80 ตัวอักษร)
7. ❌ ห้ามมีปุ่มเกิน 2 ปุ่มใน Modal เดียวกัน
8. ❌ ห้ามใช้ภาษาเทคนิคใน Body เช่น "Error 500", "null", "timeout"
9. ❌ ห้าม Title ว่างเปล่า (ยกเว้น Loading modal)
10. ❌ ห้ามใช้สีเป็น signal เพียงอย่างเดียว — ต้องมี icon shape ประกอบ (WCAG 1.4.1)
11. ❌ ห้ามใช้ icon △ แดง กับ action ที่ reversible ได้
12. ❌ ห้ามใช้คำพหูพจน์ผสม เช่น "รายการ(s)" — ใช้ "รายการทั้งหมด" หรือ "รายการนี้"

## 8. Checklist ก่อนส่งงาน

### Icon
- [ ] เลือก icon type ถูกตามประเภท Modal
- [ ] Confirmation ทั่วไป = ⚠️ เหลือง / Destructive = △ แดง
- [ ] Icon มี shape แตกต่าง ไม่ใช้สีเพียงอย่างเดียว

### Title
- [ ] ≤ 40 ตัวอักษร
- [ ] ไม่ใช้ประโยคคำถาม
- [ ] รูปแบบ "กริยา + นาม"
- [ ] ไม่มีคำว่า "ยืนยัน" นำหน้า

### Body
- [ ] ≤ 2 บรรทัด (80 ตัวอักษร)
- [ ] บอกผลที่จะเกิดขึ้น
- [ ] ไม่มีศัพท์เทคนิคหรือ error code
- [ ] ไม่ตำหนิผู้ใช้
- [ ] ใช้ "คลิก" ไม่ใช่ "กด"

### Button
- [ ] ★ แทนที่ "ยืนยัน" placeholder ด้วย verb จริงก่อน implement ← **จุดสำคัญสุด**
- [ ] Primary mirror Action ใน Title
- [ ] ปุ่มไม่เกิน 2 ปุ่ม
- [ ] เลือก layout ถูก (Double vs Single)
- [ ] Destructive = Double layout เสมอ
- [ ] Error Non-recoverable = Single / Recoverable = Double

### โทน
- [ ] ภาษาเป็นทางการระดับกลาง
- [ ] ใช้ "คลิก" ไม่ใช่ "กด"
- [ ] ใช้ภาษาไทยล้วน

## 9. วิธีใช้ไฟล์นี้กับ AI

### Prompt Pattern พื้นฐาน
```
อ้างอิง Modal UX Writing Guide ที่แนบมา

Feature: [ชื่อ feature]
Action: [สิ่งที่ผู้ใช้กำลังทำ]
ประเภท Modal: [Confirmation / Destructive / Success / Error / Info / Loading]
Reversible: [ได้ / ไม่ได้]
ข้อมูลที่ต้องแสดง: [...]

ช่วยเขียน Title, Body, Primary CTA, Secondary CTA ที่ถูกต้องตาม guide
```

### Prompt Pattern สำหรับ Review
```
อ้างอิง Modal UX Writing Guide ที่แนบมา

ตรวจสอบข้อความ modal ต่อไปนี้และแก้ให้ถูกต้องตาม guide:
Title: [...]
Body: [...]
Primary: [...]
Secondary: [...]
```

### Prompt Pattern สำหรับ Feature ใหม่ที่มีหลาย Modal
```
อ้างอิง Modal UX Writing Guide ที่แนบมา

Feature: [ชื่อ feature]
User flows ที่ต้องการ Modal:
1. [action 1] — ประเภท [...]
2. [action 2] — ประเภท [...]
3. [action 3] — ประเภท [...]

ช่วยเขียน modal copy ทั้งหมดให้ครบ พร้อมระบุ layout ที่ควรใช้
```
